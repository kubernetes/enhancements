[experience reports]: https://github.com/golang/go/wiki/ExperienceReports

### Goals
- Define OpenAPI v3 Specification that will be published by the API servers
- Improve the definition names
   - do not leak package paths
   - be prepared for $ref support (e.g. referencing a PodSpec from a CRD)
- Reuse parameters
- Reduce CPU load due to spec merging
- Align aggregated API server OpenAPI specs with per-group APIServices
- Allow aggregated API servers to publish v2 and v3 to kube-aggregator, without doubling the CPU load

### Non-Goals
 - Define a spec that uses every field from OASv3 

## Proposal
### The spec
The following example demonstrates the specification structure that will be published by the API servers. It defines two endpoints(`paths`), a set of objects(`components/schemas`) and reusable parameters(`components/parameters`) for operations e.g. `GET`.

It was created from [spec3](https://github.com/kubernetes/kube-openapi/pull/169) package that was created based on [OpenAPI Specification](https://github.com/OAI/OpenAPI-Specification/blob/master/versions/3.0.0.md).

### New names for objects

The old spec uses the full path to a package as a name for an object, for example `ConfigMap` is defined as `io.k8s.api.core.v1.ConfigMap`.
This not only doesn't add any value but also reveals the internal path to a type. 

Instead the new names will contain information about a group, version and kind(`GVK`). `ConfigMap` in the new format would become `core.v1.ConfigMap`.

### Reusable parameters

In order to make the spec smaller and more clearer common parameters for operations will be kept under `components/parameters`. 
The [OASv3 spec](https://github.com/OAI/OpenAPI-Specification/blob/master/versions/3.0.0.md#parameter-object) says that a unique parameter is defined by a combination of a name and location. 
Thus to make a parameter unique across all operations its name will be derived as `name.location.SHA1(spec3.Parameter.MarshalJSON()) | hex.EncodeToString() | hexEncodedString[1:5]`.
The last part takes the first 6 characters from a hex-encoded `SHA1` value that was calculated from a JSON encoded representation of a parameter e.g. `query.pretty.5ca316`.

### Publish specifications per API group
The API groups were primarily implemented to make it easier to extend the Kubernetes API. 
This is reflected in the REST paths where each request must be in the form of `/apis/$GROUP_NAME/...`.

Thus exposing OpenAPI specification should follow exactly the same pattern.
This proposal proposes to publish per-group spec under `apis/$GROUP_NAME/openapi/v3` as opposed to the current approach which exposes one document for all groups under `openapi/v2`

### Aggregated APIs
Since an aggregated API server can claim more that one group, the OpenAPI specifications will be published per group for exactly the same reasons as in the previous section.

### Implementation Details

- There is no need to maintain and generate separate schema definitions (`GetOpenAPIDefinitions`) for v2 and v3 specs. 
  [A new extension](https://github.com/kubernetes/kube-openapi/pull/166) `x-kubernetes-v2-spec` will be used for those types have that differences in v2 and v3 schema. 
  For now, it will be meant for `IntOrString`. In v2 spec, a workaround was made which will be changed by the use of `anyOf` for v3.
  
- In order to generate the new names for the definitions `openapi-gen` will be changed.
  For types that don't have kind e.g. `PodSpec` the golang type will be used.
  In cases, the group/version can't be determined it will be discovered the same way the [code generators do it](https://github.com/kubernetes/kubernetes/blob/master/staging/src/k8s.io/code-generator/cmd/informer-gen/generators/packages.go#L141).
  Since the new names will be in the form of `GVK` the `x-kubernetes-group-version-kind` extension will be removed from the v3 specification.
