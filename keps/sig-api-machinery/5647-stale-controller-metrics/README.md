We propose the introduction of instrumentation to figure out the staleness of
KCM controllers and report when they have not been updated for a certain amount
of time. We do this by adding probers into common codepaths that run into
scalability limits, such as statefulsets and expose metrics that give an idea of
how out of sync a controller is with the apiserver itself.

## Motivation

As Kubernetes increases in scale, greater pressure is being put on all
components. One issue that arises are controllers falling out of sync with the
apiserver as they cannot keep up. When this happens, a controller may act on
information that is old without realizing and get stuck in an irrecoverable
state where it keeps trying to act on the world and does not see its own writes
until far later. 

When situations like this arise, there is currently no way to easily tell that
the controller is overloaded. Workqueue depth may help but it isn't perfect
since we may have a short workqueue but the underlying resources have not been
caught up and we still get stale reads because of this.

### Goals

The goal of this kep is to add a set of metrics that we define for a certain set
of controllers that will expose how stale their caches are compared to the
apiserver.

### Non-Goals

Currently, there is no plan to add general metrics for every resource type. We
will define what some of the most common scale blockers are and define metrics
specifically for them. Pods are usually in this category.

We also will focus on metrics in this KEP and not propose solutions to the
issues, those can either be proposed in future KEPs or as bugfixes which may be
helped to be tracked down with the metrics proposed in this KEP.

## Proposal

We propose the exposure of several key metrics, a histogram metric for every
controller that is under high churn in high scale conditions. This metric will
follow the form 

```
&compbasemetrics.HistogramOpts{
		Subsystem:      "{my_controller_name}_controller_manager",
		Name:           "{my_controller_name}_pod_watch_delay_total_seconds",
		Help:           "Watch delay seconds",
		StabilityLevel: compbasemetrics.ALPHA,
		Buckets:        compbasemetrics.LinearBuckets(1, 1, 300),
	}
```

and will expose the overall time that the reads for pods for that controller has
lagged behind the upstream apiserver.

### User Stories (Optional)

#### Story 1

I am a cluster administrator, I want to be able to check my metrics and see if
my controllers are stale to figure out if I need more resources for my KCM
workloads. I can now see how far behind my controllers are compared to the
apiserver and decide to provision resources accordingly.

#### Story 2

I am a user and am trying to optimize my workloads, I look at my usage patterns
and try different approaches to how I deploy my workloads to reduce the
staleness of my controllers to ensure my deployments are as up to date as
possible.

### Risks and Mitigations


By adding a new prober like this, we introduce more APIServer requests to the
mix. In doing so we may add more latency when we are trying to determine the
overall latency of the system. This is unavoidable unless we want a passive
approach to our metrics. While it could possibly work, we would get large
variances in the times depending on how frequently the resources are read/
written.

## Design Details

<!--
This section should contain enough information that the specifics of your
change are understandable. This may include API specs (though not always
required) or even code snippets. If there's any ambiguity about HOW your
proposal will be implemented, this is the place to discuss them.
-->

First, we will create a prober that will run at a set time period and query
objects that are high churn and at risk of going stale. This will mostly be the
`Pod` resource, especially for Alpha. With this prober we will check at a set
interval the latest resource version. This will give us the ability to record
the exact time and resource version that a resource is reported. With the
ability to [compare the resource
version](https://github.com/kubernetes/enhancements/issues/5504), we are now
able to use inequalities to tell where a resource version may lie in this list
of resource versions we compile. We can make this as granualar as we want, with
the tradeoff of more api calls.

We will then identify controllers, (Daemonset, StatefulSet, Deployment, ...)
that are the highest churn and most at risk of running stale and will compare
the current latest read resource version to the probers. We will run through the
probers list of resource versions until we find the first object that is older
than the current read object. That will give us the time that the object in our
read cache is last updated at. With this info we will then expose metrics
showing how old our read is compared to what is actually in the apiserver.

## Alternatives
Some alternatives might be the passive detection of stale reads, however that is
prone to galse alerts and issues with periods of low churn not generating much
info. It would probably be best to approach this in a hybrid manner where we
piggy back off of requests made as much as possible but probe in periods of low
churn.
